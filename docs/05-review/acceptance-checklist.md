# MVP 验收清单

## 1. 使用范围

本清单用于两类判定：
1. **脚手架完成判定**：文档是否已达到可直接生成实现骨架的颗粒度。
2. **实现完成判定**：最小 MVP 是否已具备可运行、可验证、可演示能力。

不得混淆“文档写清楚了”和“实现已经完成了”。

## 2. 脚手架完成验收

### 2.1 必须满足
- [ ] 每个 P0 模块都能映射到页面、接口、状态机、最低脚手架文件。
- [ ] 每个关键接口都写明请求字段、响应字段、错误码、鉴权要求、审计要求。
- [ ] 每个核心模型都写明字段类型、必填、唯一性、外键、敏感性、前端暴露范围、删除策略。
- [ ] 每个关键页面都写明输入字段、触发动作、调用接口、成功跳转、失败反馈、空态/loading/阻断态。
- [ ] 每个 P0 模块都有建议测试文件、夹具、重点断言。
- [ ] 部署文档能直接回答服务、端口、挂载、env、健康检查、备份恢复。
- [ ] 冻结项均可定位到字段表、状态图、响应示例或配置表。

### 2.2 不通过判定
- [ ] 文档仍存在影响主链路实现的“待定”项。
- [ ] 某个 P0 模块仍需开发者反复追问基础契约才能开工。
- [ ] 测试或部署文档仍停留在清单级，而不是接入点级。

## 3. 实现完成验收

### 3.1 主链路验收
- [x] 用户可登录并创建 Avatar。
- [x] 用户可在 15 分钟内跑通首次体验闭环。
- [x] Persona 可生成并持久化。
- [x] Agent 可创建并执行任务。
- [x] Task 具备完整 `trace_id` 与状态流转。
- [x] 候选记忆可被确认 / 拒绝。
- [x] 高风险策略能阻断典型危险输出。
- [x] 审计日志可查询与关联。
- [ ] Docker 可一键启动演示环境。
- [x] 核心测试通过，主要安全用例覆盖。

### 3.2 通过判定

| 验收项 | 最低通过标准 |
|---|---|
| 登录与会话 | 登录成功后可访问受保护接口；未登录访问返回 `UNAUTHORIZED` |
| Avatar 管理 | 可创建、列表、详情、更新，返回字段与合同一致 |
| Persona 生成 | 导入最小样本后可生成 Persona，并可切换当前激活版本 |
| Agent 执行 | 可创建默认 Agent、启停 Agent、成功发起任务 |
| Task 状态流转 | 至少覆盖 `pending -> running -> succeeded/failed/blocked` |
| Memory 确认链路 | 同一 memory 不允许重复 confirm / reject，冲突返回 `MEMORY_STATE_CONFLICT` |
| Policy 阻断 | 至少验证 3 类高风险场景被阻断并写入审计 |
| Audit 追踪 | 可通过 `trace_id` 串联任务、策略命中、结果状态 |
| Docker 演示环境 | 使用单一 Compose 命令可启动最小演示环境 |
| 自动化测试 | 主链路具备 unit、integration、最小 e2e；安全阻断具备专项用例 |
| 分身成长报告合同 | 术语、数据来源、状态语义、证据链要求在文档中冻结且彼此一致 |

## 3.3 分身成长报告设计验收
- [x] Growth Report 统一使用 `Avatar Growth Report` / 分身成长报告术语。
- [x] 文档明确规定只有 `confirmed` memory 才能进入“系统已学会”的结论。
- [x] 文档明确区分 Persona、候选记忆、已确认长期记忆、审计证据。
- [x] Growth Report 至少定义 `empty` / `building` / `ready` / `demo` 状态。
- [x] 每条成长洞察都要求可回溯到 memory、task 或 `trace_id`。
- [x] Demo Avatar 的成长报告被明确标识为示例/seeded data。
- [x] 推荐动作仍围绕当前 MVP 主链路，不引入开放插件或高自治多代理承诺。

## 4. 不通过判定

出现以下任一情况，不得宣告 MVP 实现完成：
- 主链路仍需人工改库或手工改请求才能跑通。
- API 文档与实际返回结构不一致。
- Task 或 Memory 状态机存在未定义分支。
- 高风险输出无法被阻断或阻断后无审计记录。
- Docker 环境无法稳定启动演示链路。
- 关键测试仍处于“待补”而非已执行状态。

## 5. 验收证据要求

| 验收类型 | 最低证据 |
|---|---|
| 脚手架完成 | 对应文档章节、字段表、状态图、建议文件路径 |
| 接口实现 | 接口返回示例、错误码验证记录、OpenAPI 导出 |
| 测试执行 | 测试结果、失败记录、覆盖矩阵 |
| 部署可用 | Compose 启动记录、`/health` 结果、smoke 测试结果 |
| 安全阻断 | 阻断样例、错误响应、审计记录 |

## 6. 使用规则

- 只有“脚手架完成验收”通过后，才能宣告文档达到 scaffold-ready。
- 只有“实现完成验收”通过后，才能宣告 MVP 完成。
- 若冻结项变化，必须重新执行相关验收。
- 定版后的验收标准原则上不得修改；仅允许明显错漏、重大矛盾修复，或有证据支持的显著先进性、可落地性、科学性提升。
