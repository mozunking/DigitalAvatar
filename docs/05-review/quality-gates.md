# 质量门禁

## 1. 门禁目标

1. 把“文档可读”提升为“文档可生成脚手架、实现可验证、发布可回滚”。
2. 防止在冻结项未明确时启动并行实现。
3. 防止在测试、部署、安全证据不足时宣告完成或发布。
4. 定版后将关键文档标记为 frozen baseline，原则上不再修改。

## 2. 文档门禁

### 2.1 scaffold-ready 门禁

| 检查项 | 通过标准 |
|---|---|
| 产品规格 | 每个 P0 模块都映射到页面、接口、状态机、最低脚手架文件 |
| 用户旅程 | 每步都写明字段、动作、接口、跳转、失败反馈、空态/loading/阻断态 |
| API 合同 | 每个关键接口写明请求字段、响应字段、错误码、鉴权要求、审计要求 |
| 数据模型 | 每张核心表写明字段、类型、唯一性、外键、敏感性、前端暴露范围、删除策略 |
| 安全架构 | token 生命周期、权限矩阵、策略阶段、阻断响应、加密矩阵明确 |
| 前端架构 | 路由表、目录骨架、页面与 store/api/types 映射明确 |
| 测试计划 | 每个 P0 模块有建议测试文件、夹具、断言重点 |
| 部署计划 | 服务、端口、挂载、env、健康检查、备份恢复明确 |

### 2.2 不通过判定
- 任一关键文档仍停留在原则/清单级，无法直接指导骨架生成。
- 任一冻结项无法定位到字段表、状态图、示例响应或配置表。
- 文档间出现冲突，且未有裁决记录。

## 3. 冻结基线门禁

### 3.1 frozen baseline 门禁
- `requirements.md`、`functional-spec.md`、`user-journeys.md`、`api.md`、`data-model.md`、`security-architecture.md`、`frontend-architecture.md`、`observability.md`、`development-plan.md`、`testing-plan.md`、`deployment-plan.md`、`release-plan.md` 已完成交叉一致性检查。
- `interface-freeze-checklist.md` 全部冻结项可提供证据。
- `acceptance-checklist.md` 区分脚手架完成验收与实现完成验收。
- `document-map.md` 已标记 frozen baseline 范围。
- `decision-log.md` 已记录冻结策略。
- `CONTRIBUTING.md` 与 `skill.md` 已写明冻结后变更规则。

### 3.2 冻结后变更例外
仅允许以下情况修改 frozen baseline：
1. 明显错漏。
2. 重大矛盾修复。
3. 有证据支持的显著先进性、可落地性、科学性提升。

任何例外修改都必须同步更新：
- `decision-log.md`
- `interface-freeze-checklist.md`
- 受影响的方案文档
- 受影响的验收与评审文档

## 4. 实现门禁

| 检查项 | 通过标准 |
|---|---|
| 合同一致性 | API、状态机、错误码、权限边界与文档一致 |
| 主链路可运行 | 登录、Avatar、Persona、Agent、Task、Memory、Audit 可串联 |
| 安全边界 | 未授权、越权、高风险输出能被阻断 |
| 可观测性 | `/health`、日志字段、`trace_id` 透传、审计记录齐备 |
| 部署接入 | Compose、env、脚本、备份恢复与文档一致 |

## 5. 测试门禁

| 检查项 | 通过标准 |
|---|---|
| unit | 关键状态机与服务逻辑可执行 |
| integration | 核心 API 覆盖登录、Avatar、Persona、Agent、Task、Memory、Audit |
| e2e | 至少覆盖首次引导和单任务闭环 |
| security | 至少 3 类高风险阻断用例通过 |
| smoke | Compose 环境可执行最小健康检查与主链路冒烟 |

出现以下任一情况不得通过：
- 测试仍处于“计划中”而不是“已执行”。
- 关键状态机缺少异常路径测试。
- 安全阻断未落地自动化验证。

## 6. 发布门禁

- 演示环境可重复启动。
- `/health` 返回正常，关键服务可连通。
- 发布前检查单完成。
- 回滚条件明确，恢复路径可验证。
- 发布说明、已知问题、演示脚本齐备。

## 7. 评审门禁

评审必须覆盖：
- 用户视角：是否易理解、易上手、主价值清楚。
- 产品视角：是否范围稳定、主链路闭环、P0/P1 边界清晰。
- 开发视角：是否能直接生成骨架并并行开工。
- 测试视角：是否有清晰验证入口和证据。
- 安全视角：是否明确认证、权限、隐私、阻断规则。
- 运维视角：是否具备部署、健康检查、日志、备份恢复。

## 8. 门禁结果分类

| 结果 | 含义 | 后续动作 |
|---|---|---|
| pass | 门禁通过 | 可进入下一阶段 |
| conditional pass | 存在非阻断缺口 | 补齐指定项后复核 |
| fail | 存在阻断性缺口 | 禁止进入下一阶段 |

## 9. 使用规则

- 未通过 scaffold-ready 门禁，不得进入多代理实现。
- 未通过 frozen baseline 门禁，不得标记文档定版。
- 未通过实现/测试/发布门禁，不得宣告 MVP 完成或发布。
- 若某门禁无法回答“证据在哪里”，视为未通过。
