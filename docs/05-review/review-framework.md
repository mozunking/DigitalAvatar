# 评审框架

## 1. 评审目标

1. 评审不只看“写了没有”，而要看是否足以直接驱动脚手架生成、实现落地和发布验证。
2. 评审必须覆盖用户、产品、开发、测试、安全、运维六个视角。
3. 评审结论必须能落到具体文档修改、门禁结果或冻结裁决，不能停留在抽象意见层。

## 2. 评审视角

- 用户视角：是否易理解、易上手、价值明确、首次体验闭环清晰。
- 产品视角：是否范围清晰、闭环完整、优先级与边界稳定、指标可量化。
- 开发视角：是否边界清楚、契约稳定、能并行实现、能直接生成骨架。
- 测试视角：是否具备可验证的验收入口、夹具、断言重点与失败判定。
- 安全视角：是否明确认证、权限、隐私、敏感字段处理、输出风险边界。
- 运维视角：是否考虑部署、日志、监控、健康检查、备份恢复、回滚。

## 3. 评审方法

1. 先检查是否覆盖主链路。
2. 再检查是否存在范围膨胀、重复文档或职责混写。
3. 再检查冻结项是否能直接指导并行开发。
4. 检查每个角色是否都能找到自己的输入、输出、依赖与验收标准。
5. 检查文档是否达到 scaffold-ready，而不只是概念完整。
6. 最后检查验收、整改、标准化与冻结治理是否闭环。

## 4. 强制评审维度

| 维度 | 必须回答的问题 |
|---|---|
| 主链路完整性 | 用户能否从登录走到任务执行、记忆确认、审计查看？ |
| 页面可生成性 | 每个关键页面是否能直接生成 view/store/api/components 骨架？ |
| 接口可生成性 | 每个关键接口是否能直接生成 route/schema/service/client/types？ |
| 模型可生成性 | 每张核心表是否已具备字段、约束、敏感性、删除策略？ |
| 测试可接入性 | 每个 P0 模块是否有建议测试文件、夹具、断言重点？ |
| 部署可落地性 | 是否能直接生成 Compose、env、脚本、smoke？ |
| 治理可执行性 | 冻结规则、变更例外、评审记录、决策索引是否齐备？ |

## 5. 评审输出格式

每次正式评审至少输出：
- 评审范围
- 参与视角 / 角色
- 主要优点
- 阻断性问题
- 非阻断改进项
- 结论（pass / conditional pass / fail）
- 要求修改的目标文档
- 是否影响 frozen baseline

## 6. 阻断性问题判定

出现以下任一情况，评审结果必须为 `fail`：
- 主链路未闭环。
- 关键接口、模型、状态机仍存在“待定”。
- 文档无法直接支持多代理并行开工。
- 安全边界没有明确阻断语义。
- 测试和部署仍停留在概念级，无法生成接入骨架。
- 关键文档间存在冲突且无裁决。

## 7. 本轮定版评审结论模板

### 优点
- 文档分层明确，治理 / 产品 / 架构 / 交付 / 评审边界清楚。
- P0 模块已被映射到页面、接口、状态机、测试与部署接入点。
- 开发计划、测试计划、部署计划已具备脚手架生成输入能力。

### 必查项
- 冻结项是否都有证据。
- 评审、验收、门禁、治理是否已同步升级。
- 是否已明确“定版后原则上不改”的例外规则。

## 8. 使用规则

- 评审后必须把问题落到明确文档修改，而不是停留在意见层。
- 若评审结论影响冻结项，必须同步更新 `decision-log.md`、`interface-freeze-checklist.md` 与相关文档。
- 若评审无法回答“开发智能体是否还需要追问基础契约”，则不得判定为 scaffold-ready。
