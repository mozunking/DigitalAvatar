# 标准化执行方法

## 1. 文档体系建设方法

1. 先识别主源文档与历史输入。
2. 按治理 / 产品 / 架构 / 交付 / 执行 / 评审 / 改进分层拆解。
3. 保留历史输入，不把历史评审直接复制为现行规则。
4. 所有冻结项必须有索引入口、证据定位和变更同步规则。
5. 每份关键文档都要写到可支持脚手架生成，而不是停留在说明层。

## 2. 写到 scaffold-ready 的方法

对关键文档，至少补齐以下内容：
- 产品文档：模块、页面、接口、状态机、最低脚手架文件映射
- 用户旅程：步骤、字段、动作、接口、跳转、失败反馈、空态/loading/阻断态
- API 文档：path/query/body/response 字段、错误码、鉴权、审计要求
- 数据模型：字段、类型、唯一性、外键、敏感性、前端暴露范围、删除策略
- 前端文档：路由、目录骨架、页面与 store/api/types/components 映射
- 测试文档：建议文件、夹具、种子数据、断言重点
- 部署文档：服务、端口、挂载、env、健康检查、备份恢复

判断标准：若开发智能体仍需追问“字段是什么、页面怎么跳、接口怎么鉴权、测试怎么接、Compose 怎么配”，则还未到 scaffold-ready。

## 3. 并行开发准备方法

1. 先冻结领域模型、状态枚举、API、错误码、审计结构。
2. 再做多代理职责拆解。
3. 每个代理都要定义输入、输出、依赖和验收标准。
4. 每个代理都要能回答首批创建哪些文件和目录。
5. 冲突由主控角色统一裁决。

## 4. 文档评审方法

1. 从用户、产品、开发、测试、安全、运维六个视角检查。
2. 重点识别：是否可执行、是否可验证、是否可生成脚手架、是否有重复、是否范围失控。
3. 评审后必须把问题落到明确文档修改，而不是停留在意见层。
4. 若关键文档之间有冲突，先裁决再继续细化，不能带着矛盾冻结。

## 5. frozen baseline 方法

1. 关键文档完成 scaffold-ready 门禁后，进入 frozen baseline 候选。
2. 完成交叉一致性检查、验收/门禁/治理同步后，才可正式定版。
3. 定版后原则上不再修改。
4. 仅允许以下例外：
   - 明显错漏
   - 重大矛盾修复
   - 有证据支持的显著先进性、可落地性、科学性提升
5. 任何例外修改都必须同步更新决策日志、冻结清单、相关方案与评审文档。

## 6. GitHub 风格仓库最小文档面

1. 根目录必须有 README、CONTRIBUTING、SECURITY、PRIVACY、LICENSE、CLAUDE。
2. `docs/` 必须有总入口与阅读顺序。
3. 方案、执行、检查、改进必须分层，避免混写。
4. Issue / PR 模板要能直接约束贡献输入质量。

## 7. 使用约束

- 不要为了“看起来完整”而堆砌抽象原则，优先补可执行输入。
- 不要在 frozen baseline 里引入未验证的大范围扩展。
- 若某标准无法转化为目录、字段、状态、接口、用例或配置，说明还不够落地。
