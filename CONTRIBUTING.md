# Contributing to Digital Avatar

## 贡献目标

本仓库当前已包含可运行 MVP 实现与成体系文档。贡献应优先提升以下内容：

- 产品方案的可落地性
- 技术契约的清晰度与一致性
- 多代理并行开发所需的职责边界与交付标准
- 评审、验收、整改与治理机制
- 真实实现、测试、部署与文档的一致性

## 开始之前

提交贡献前，请先阅读：

1. [README.md](README.md)
2. [docs/README.md](docs/README.md)
3. [docs/00-governance/document-map.md](docs/00-governance/document-map.md)

## frozen baseline 规则

以下文档属于默认冻结基线，原则上不得随意修改：
- 产品、架构、交付、评审、治理中被 `document-map.md` 标记为 frozen baseline 的文档

仅允许以下情况修改 frozen baseline：
1. 明显错漏。
2. 重大矛盾修复。
3. 有证据支持的显著先进性、可落地性、科学性提升。

任何此类修改都必须在 PR 中明确说明：
- 为什么必须改
- 证据是什么
- 影响哪些文档、角色、门禁或冻结项
- 是否需要同步更新 ADR / 决策日志 / 验收文档

## 贡献范围

欢迎贡献：

- 文档结构优化
- 需求澄清与验收标准补充
- 架构、接口、数据模型、安全方案的细化
- 测试、部署、评审与治理流程完善
- GitHub 模板与协作规范完善
- 不破坏 frozen baseline 前提下的可落地性增强

当前不接受：

- 偏离既定 MVP 冻结范围的大规模功能扩张
- 未经过文档同步更新的重大契约变更
- 将历史评审内容直接复制为新规范而不做整理
- 未提供证据的 frozen baseline 变更

## 提交流程

1. 明确你的变更属于哪个文档域：产品、架构、交付、执行、评审、改进、治理。
2. 先更新对应文档，再更新索引或映射文件。
3. 如果变更涉及冻结项、契约、接口或安全边界，必须同步更新：
   - `docs/00-governance/decision-log.md`
   - `docs/03-delivery/interface-freeze-checklist.md`
   - `docs/05-review/quality-gates.md`
   - 相关 ADR 或专题文档
4. 如果变更影响验收、测试、部署或发布，必须同步更新对应交付/评审文档。
5. 提交 Pull Request，并说明：
   - 为什么要改
   - 改动影响哪些文档或角色
   - 是否改变了冻结项、验收口径或执行顺序
   - 如何验证这次修改仍保持一致性

## Pull Request 要求

每个 PR 至少说明：

- 变更背景
- 主要改动
- 影响范围
- 校验方式
- 后续待办

如涉及重大决策，请附上相应 ADR 或指向对应文档。

## 文档编写要求

- 优先写可执行规则，不写空泛原则
- 优先写能直接驱动脚手架生成的字段、状态、目录、接口、用例、配置
- 避免重复复制已有大段内容，使用链接和引用建立文档网络
- 对范围、边界、输入、输出、负责人、验收标准要写清楚
- 对 P0 / P1 / P2 优先级要明确
- 历史输入与现行规范分开维护
- 若文档无法回答“开发者该创建什么、如何验证、谁负责”，则说明颗粒度仍不足

## 分支与提交建议

当前仓库尚未形成固定开发分支策略。建议在实施期采用短分支协作：

- `docs/...`：文档体系建设与修订
- `feat/...`：代码功能实现
- `fix/...`：问题修复
- `chore/...`：治理、模板、脚手架类变更

提交信息建议聚焦“为什么”，避免只写“update docs”。

## 评审标准

文档类 PR 应至少通过以下检查：

- 能否被目标角色直接执行
- 是否与现有冻结项冲突
- 是否补足了问题，而不是制造新的歧义
- 是否同步更新导航、索引、验收或整改文档
- 若涉及 frozen baseline，是否满足例外条件并附证据
