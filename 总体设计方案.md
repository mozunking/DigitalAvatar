# 1. 项目概述

## 1.1 项目名称
**Digital Avatar**

## 1.2 项目 Slogan
**你的开源数字孪生分身，复刻你的风格，辅助你执行任务，持续沉淀你的个人智慧。**

## 1.3 项目定义
Digital Avatar 是一个以“数字孪生分身”为核心的开源智能体系统。
系统围绕用户的风格、偏好、知识、行为习惯和任务执行方式进行建模，生成一个可控、可审计、可持续成长的个人智能分身。

它不是单纯的聊天机器人，也不是只做工具编排的 Agent Shell，而是一个强调：

- **风格复刻**
- **任务辅助执行**
- **个性化记忆沉淀**
- **本地优先与数据主权**
- **可审计、可扩展、可持续迭代**

的个人智能系统。

## 1.4 开源协议建议
V12 版本建议统一采用：

- **Apache 2.0**

原因：

1. 降低协议理解成本
2. 降低企业接入顾虑
3. 降低插件生态扩展复杂度
4. 避免首版混用 MIT / GPL / Apache 带来的边界争议

# 2. 版本目标与边界

## 2.1 48小时 MVP 目标
48 小时版本只验证一个核心闭环：

> 用户能导入自己的资料/对话样本，生成初始数字分身，并让一个受控子代理按其风格完成一个任务，同时自动捕捉候选记忆，经过用户确认后写入记忆库。

## 2.2 48小时 MVP 范围

### 必须完成
1. 分身创建
2. 用户资料/对话样本导入
3. Persona / Profile 生成
4. 单子代理创建
5. 单任务执行
6. 候选记忆自动捕捉
7. 候选记忆确认入库
8. 基础策略校验
9. 审计日志记录
10. Web 控制台
11. Docker 一键部署
12. 基础测试与文档

### 暂不纳入 MVP
1. Electron 正式桌面端
2. 全平台安装包
3. 插件市场
4. 微信/钉钉接入
5. 多设备同步
6. MBTI 系统
7. 双轨传承系统
8. 永续执行模式
9. 企业级监控告警
10. 完整开放插件生态

## 2.3 7天 Beta 目标
在 MVP 基础上，形成一个可对外展示与试用的 Beta 版本，重点增加：

1. 多模型 Provider 抽象
2. 多类记忆模型
3. 多子代理模板
4. 插件最小机制
5. 数据导出/备份
6. 更完整策略规则
7. Docker + Web 稳定交付
8. 可选桌面端实验版

# 3. 产品价值与设计原则

## 3.1 核心价值主张
Digital Avatar 提供三类价值：

### 1）个性化表达复刻
系统能学习用户的表达风格、任务习惯、偏好约束与价值边界。

### 2）可控任务执行
系统不是无边界自治，而是在可控权限和可审计链路下，协助执行明确任务。

### 3）长期记忆沉淀
系统能把用户真实有价值的信息从对话和任务中提炼出来，经确认后沉淀为长期资产。

## 3.2 核心原则
1. **先跑通闭环，再扩展生态**
2. **先定义边界，再并行开发**
3. **先保证可控，再追求复杂能力**
4. **默认本地优先，默认可审计**
5. **MVP 聚焦核心价值，不追求全量形态**
6. **关键主链路用服务接口，扩展能力用事件机制**
7. **自动化优先，但高风险变更必须设门禁**

# 4. AI 多智能体开发模式

## 4.1 开发组织方式
项目采用：

- 1 个主开发智能体
- 6~8 个专项开发子代理
- 1 个测试与校验智能体

## 4.2 两阶段冻结机制

### 第一阶段：0-2小时冻结主骨架
冻结内容：

- 仓库结构
- 核心实体定义
- API 路由骨架
- 错误码规范
- 事件命名规范
- 测试目录结构
- 提交规范
- 代码风格规范

### 第二阶段：2-12小时补充冻结细节
补充内容：

- 字段级 Schema
- 审计日志结构
- Persona 输入输出结构
- Memory 状态流转
- 前端页面状态机
- Policy 规则定义

## 4.3 角色拆分

| 角色 | 核心职责 | 主要交付 |
|------|----------|----------|
| 主开发智能体 | 架构骨架、规范冻结、集成发布 | 仓库骨架、架构文档、集成版本 |
| 内核开发子代理 | Agent 调度、Provider 抽象、运行时骨架 | Core 服务 |
| Persona/Memory 子代理 | Persona 建模、记忆捕捉、检索与确认 | Memory 服务 |
| Policy/Audit 子代理 | 规则校验、输出检查、审计日志 | Policy 模块 |
| API 与集成子代理 | OpenAPI、DTO、接口服务、Mock 合同 | API 层 |
| 前端控制台子代理 | Web 交互流程与页面实现 | Web 控制台 |
| 测试子代理 | 单测、集成、E2E、回归与报告 | 测试套件 |
| 文档与发布子代理 | README、部署文档、CI 配置、发布说明 | 文档与流水线 |

# 5. 总体架构设计

## 5.1 分层架构

```text
┌──────────────────────────────────────────────┐
│ 应用层 Application Layer                     │
│ Web Console / OpenAPI / CLI / Docker        │
├──────────────────────────────────────────────┤
│ 领域层 Domain Layer                         │
│ Avatar域 | Agent域 | Memory域 | Policy域    │
├──────────────────────────────────────────────┤
│ 基础设施层 Infrastructure Layer             │
│ LLM Provider | DB | Vector Store | Audit    │
├──────────────────────────────────────────────┤
│ 运行与扩展层 Runtime & Extension            │
│ Event Bus | Module Loader | Task Worker     │
└──────────────────────────────────────────────┘
```

## 5.2 架构约束
1. 核心主链路优先走同步服务接口
2. 扩展通知、异步处理使用事件总线
3. 策略校验嵌入关键入口，而不是依赖单点模块
4. 模块边界按领域划分，不按“页面”划分
5. 数据模型优先稳定，UI 交互可迭代

# 8. 策略与审计体系

## 8.1 设计原则
V12 不再采用模糊的“三大定律硬编码”说法，统一改为：

> **Policy & Audit Layer（策略与审计层）**

## 8.2 策略分层
策略分三类：

### 1）系统安全策略
用于约束工具调用、越权操作、危险执行。

### 2）法律/风险策略
用于拦截高风险输出、敏感内容、冒充和误导。

### 3）用户边界策略
用于尊重用户自身偏好、限制、拒绝事项。

## 8.3 规则结构示例

```yaml
policy_id: P-001
policy_code: block-high-risk-impersonation
name: 禁止冒充真人进行高风险承诺
scope: outbound
severity: critical
condition:
  type: expression
  expr: "impersonation == true && risk_level == 'high'"
action: block
message: "当前内容触发高风险冒充规则，已阻断输出。"
enabled: true
```

## 8.4 审计原则
系统默认开启审计。

审计记录至少包含：

- 触发者
- 动作
- 资源对象
- 输入摘要
- 输出摘要
- 命中策略
- 结果状态
- trace_id
- 时间戳
- 哈希链字段

## 8.5 审计表述
项目文档统一使用：

> 采用哈希链和校验机制，支持篡改检测与审计追溯。

不使用“绝对不可篡改”的表述。

# 9. 记忆系统设计

## 9.1 目标
让系统“越用越懂你”，但不把所有对话原样塞进数据库，而是：

- 先提炼
- 再分类
- 再等待确认
- 最后入库沉淀

## 9.2 记忆类型

### 1）Profile Memory
稳定偏好、风格、约束。

示例：
- 回答喜欢先结论后展开
- 偏好中文简洁表达
- 不接受擅自替自己做承诺

### 2）Episodic Memory
具体任务或事件的摘要。

示例：
- 今天完成了方案评审
- 用户否定了某个设计方向

### 3）Procedural Memory
做事流程和方法。

示例：
- 写文档先出目录
- 评审时先给结论再提改进建议

### 4）Knowledge Memory
长期知识片段或稳定观点。

示例：
- 某项目背景、术语定义、业务上下文

## 9.3 记忆状态流转

```text
captured -> pending_confirm -> confirmed -> archived
                      └-> rejected
```

## 9.4 写入规则
1. 自动捕捉默认进入 `pending_confirm`
2. 高敏感信息必须人工确认
3. 低价值信息可直接丢弃
4. 用户可纠正、删除、归档
5. 每条记忆必须有类型和来源

## 9.5 检索规则
建议采用组合检索：

1. 类型过滤
2. 标签过滤
3. 时间窗口过滤
4. 向量检索
5. 重排序
6. token budget 裁剪

# 10. 插件与扩展机制

## 10.1 MVP 范围
MVP 仅支持：

- 官方模块
- 白名单模块
- 本地注册模块

不开放任意第三方插件市场。

## 10.2 Manifest 规范

```json
{
  "id": "memory-capture",
  "name": "Memory Capture Module",
  "version": "0.1.0",
  "entry": "main.py",
  "description": "Capture candidate memories from task execution results.",
  "permissions": [
    "memory:write",
    "event:subscribe",
    "audit:append"
  ],
  "hooks": [
    "on_task_completed",
    "on_response_generated"
  ],
  "config_schema": {
    "type": "object",
    "properties": {
      "min_confidence": {
        "type": "number",
        "default": 0.7
      }
    }
  }
}
```

## 10.3 模块生命周期
- install
- enable
- disable
- unload
- remove

## 10.4 模块运行约束
MVP 阶段建议：

- 不开放任意系统命令执行
- 不开放任意文件路径访问
- 模块权限通过 manifest 声明
- 未授权权限默认拒绝

# 16. 代码规范与工程约束

## 16.1 后端规范
1. Python 3.11+
2. 所有公开函数必须带类型注解
3. Pydantic 统一 DTO
4. 服务层与路由层分离
5. 不允许把业务逻辑堆在 API Controller

## 16.2 前端规范
1. TypeScript strict 模式
2. API 类型自动生成或集中维护
3. 页面状态与领域对象解耦
4. 组件按页面域拆分

## 16.3 通用规范
1. 错误码统一定义
2. 关键函数必须有 docstring
3. 外部调用必须有异常处理
4. 不允许未声明权限的模块访问敏感能力
5. 所有 PR 必须通过 lint 与测试

---

# 17. 测试与验收标准

## 17.1 单元测试标准
- 核心服务覆盖率 ≥ 85%
- Memory 域关键逻辑 ≥ 90%
- Policy 模块 ≥ 90%

## 17.2 集成测试标准
必须覆盖以下流程：

1. 创建分身
2. 导入样本
3. 生成 Persona
4. 创建子代理
5. 执行任务
6. 捕捉候选记忆
7. 用户确认记忆
8. 查询审计日志

## 17.3 E2E 测试标准
至少覆盖 3 条主路径：

1. 初始化与分身创建流程
2. 单任务执行流程
3. 候选记忆确认流程

## 17.4 安全基线
- 无硬编码密钥
- 依赖扫描无 Critical
- 审计日志默认开启
- 默认不开启危险工具调用
- 文件访问必须白名单限制

# 23. 风险说明与决策记录

## 23.1 明确不做的“绝对化承诺”
本版本不使用以下表述：

- 100% 不可绕过
- 100% 测试覆盖
- 全平台 100% 兼容
- 48 小时交付完整生态系统

## 23.2 已做出的关键收敛决策
1. 首版只做 Web，不强推桌面端
2. 首版只优先支持 Ollama
3. 首版只做单任务主链路
4. 首版不开放任意插件生态
5. 首版统一 Apache 2.0 协议
6. “合规盾”重构为 Policy & Audit Layer
7. 记忆默认先进入候选确认，不直接全量入库
