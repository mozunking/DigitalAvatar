# Security Policy

## 安全定位

Digital Avatar 的核心承诺之一是安全可控。当前仓库仍处于方案冻结与实施准备阶段，因此安全文档既用于漏洞报告，也用于明确设计边界。

## 报告安全问题

如发现安全漏洞，请不要公开提交包含可利用细节的 Issue。

建议通过私密渠道向维护者报告，报告内容至少包括：

- 问题描述
- 影响范围
- 复现条件
- 潜在风险等级
- 建议修复方向

在尚未建立专用邮箱前，请通过私密方式联系仓库维护者，并避免公开披露敏感细节。

## 当前安全冻结项

MVP 阶段必须满足：

- JWT + Refresh Token 认证方案
- bcrypt 密码哈希
- 默认拒绝、显式授权、最小权限
- 高风险操作门禁
- LLM 输出安全过滤
- 敏感记忆默认不自动入长期库
- 审计日志与结构化日志分离，并通过 trace_id 关联
- 用户数据导出、删除、保留期限机制

详见：

- `docs/02-architecture/security-architecture.md`
- `docs/05-review/security-checklist.md`
- `docs/06-improvement/improvement-plan.md`

## 不在 MVP 范围内但需说明的风险

以下能力在 MVP 中不开放或不完全开放：

- 任意插件动态执行
- 高自治多代理编排
- 企业级 IAM / 多租户权限体系
- 企业级扩展沙箱运行时

任何试图绕过这些边界的实现，都应视为重大安全变更并单独评审。

## 安全变更要求

以下变更必须同步更新安全文档并接受专项评审：

- 认证与授权流程变更
- 数据存储与加密策略变更
- 输出安全、内容过滤、风险分级策略变更
- 隐私接口与数据生命周期规则变更
- 模块、扩展、工具调用权限边界变更

## 发布前安全检查

在代码实现落库后，最低要求包括：

- 未授权访问阻断检查
- 越权操作阻断检查
- 高风险输出阻断检查
- 敏感数据处理与脱敏检查
- 用户导出/删除接口校验
- 审计与 trace_id 完整性检查
